What both datasets agree on

  • surface_total_in_m2 is unusable in both — we fall back to surface_covered_in_m2.
  • floor and expenses are too sparse for any analysis in either dataset.
  • Roughly half of listings have no geocoding in both Rent (~53%) and Sell (~56%).
  • About 5–6% of listings have no price → dropped from Silver.
Sell has slightly more missing data overall, but the pattern is the same — one cleaning pipeline fits both.
Cleaning · Silver · step 1 / 5

Break the location string into country, region, city

Use case

The raw dataset packs the full location into a single piped string. To compare prices across regions or cities, we need each level of the hierarchy as its own column.

Before

|Brasil|São Paulo|Barueri|Alphaville|

What we did

Split that string by | into three new columns: country, region, city.

After

countryregioncity
BrasilSão PauloBarueri

Region becomes the main grouping axis for every chart that follows.

Cleaning · Silver · step 2 / 5

Drop listings that aren’t in Brazil

Use case

The dataset is advertised as Brazilian, but a handful of US listings sneak in (Miami, mostly — country = "Estados Unidos"). Mixing a couple hundred USD-denominated US listings with hundreds of thousands of Brazilian ones distorts every per-region and per-type aggregate.

What we did

  • Kept only rows where country == "Brasil".
  • Removes the Miami / “Estados Unidos” entries that otherwise show up as their own “region” in the Gold charts.

Tiny in row count, but they were skewing the regional breakdowns disproportionately.

Cleaning · Silver · step 3 / 5

Keep only the columns we’ll actually use

Use case

The raw dataset has 24 columns, but most of them are either redundant (already represented elsewhere), nearly always empty, or simply not relevant for a pricing analysis. Carrying them around just slows everything down and adds visual noise.

What we dropped — and why

  • Redundant — original location strings (already split into country / region / city) and coordinate strings.
  • Too sparse to be useful — the geonames identifier and the “expenses” field, mostly empty.
  • Not analytical — listing URLs, free-text descriptions and titles, thumbnail images.
  • Replaced — the original local-currency price (we recompute a clean USD price later).

Net result: 24 columns → 16, memory footprint roughly a third lighter.

Cleaning · Silver · step 4 / 5

Drop listings with no price

Use case

A real-estate pricing analysis can’t use a listing whose price is unknown. The rest of the missing values (surface, floor, rooms, geocoding) we leave as NaN — pandas aggregations skip them automatically, which keeps averages honest.

What we did

  • Dropped listings with no price via dropna(subset=["price"]).
  • Left every other NaN untouched — no zero-imputation, so means like price_usd_per_m2.mean() aren’t biased downward by fake zeros.

Rent: 97,353 → 92,332 Sell: 872,672 → 819,298

About 5–6% of listings removed in each dataset.

Cleaning · Silver · step 5 / 5

USD prices come pre-computed — we just use them

Use case

The original prices are in Brazilian Reais (BRL). To compare regions, years, and the rent vs sell datasets on the same scale, every chart in the deck works off USD values.

What we did

  • Used the source columns price_aprox_usd and price_usd_per_m2 as-is — Properati already converted them at the listing's own moment in time.
  • No flat-rate recompute: BRL/USD swung from ~2 to ~4 across 2013–2016, so a single hard-coded rate would distort cross-year comparisons.
  • Saved the cleaned table to the Silver layer, ready for Gold analysis.

What this analysis is not yet

Data quality

  • No outlier removal A few extreme listings can dominate regional averages. The boxplot view exposes the spread, but downstream means still include the tails.
  • Source FX We rely on Properati's pre-computed price_aprox_usd — whatever rate they used per listing, we accept as given.
  • Missing geocoding Roughly half the listings have no lat/lon — any future geographic analysis would simply lose them.

Scope

  • Snapshot Dataset frozen at 2016-11-01 — results are historical, not current market.
  • Surface surface_total_in_m2 (~84% missing) discarded; we only use covered surface.
  • Correlation Rooms vs price scatter was only run for São Paulo / houses, and not exported as a chart — not generalized to other regions or types.
  • Stats No confidence intervals, no significance tests, no holdout.
Suggested analyses

Useful next analyses to explore

The current Gold layer covers four solid views. Here are concrete extensions that would unlock new questions, using only the columns we already have.

Rent-yield map

Combine both datasets: divide the average annualised rent by the average sell price, per region (and per property type).

Answers: “Where does a landlord recover their investment fastest?”

Price-to-rent ratio

The inverse view: sell price ÷ annual rent, per region. A common over- / under-valuation indicator used in real-estate research.

Answers: “Is buying or renting cheaper, region by region?”

Top-N cities per region

Drill from region down to city: list the 10 most expensive cities inside São Paulo, Rio, Minas Gerais, etc.

Answers: “Where exactly is the premium concentrated?”

Room-count premium

Per region and property type, plot average price by number of rooms (1, 2, 3, 4+). Quantifies the marginal cost of an extra bedroom.

Useful for buyers planning families and for investors.

Geographic heatmap

For listings that have real coordinates, plot a heatmap of price-per-m².

Highlights intra-region hotspots that an aggregate hides.
